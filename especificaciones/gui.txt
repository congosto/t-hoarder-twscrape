Prototipo visual

Parte superior:
logo de la app (logo_t-hoarder.png) nombre de la app (t-hoarder-twscraper)
un separador de zona

una lista de botones horizontales con las siguientes funciones

- Settings
- Project
- Download
- Tools
- Graphs
- Charts

En la parte izquierda (20% del espacio): formulario de entrada de datos según la opción

  New Account | Active Accounts | Delete Account

     New Account
       Username
       Password
       Email
       Email Password
       Auth Token
       ct0
     Active Accounts

     Delete Account

- Project

  Select project | New project | Active projects | Desactive project

  Select project
    Project name
  New project
    Project name
  Active projects
    List of active projects
  Desactive project
    Project name
    acciones:
    - No borra el proyecto: lo mueve de data/{project} a data/desactivated/{project}, para no saturar el desplegable de proyectos activos sin perder los datos

- Download

  Search | User TL | Retweets | Comments | Advanced Comments

  Search
    Prefix (File prefix for results)
    Botón "Cargar contexto": si el Prefix ya tiene una descarga previa, rellena
      Query/Product/From/To/Frequency con lo guardado; si no tiene contexto,
      limpia esos campos a sus valores por defecto
    Query (Twitter query, supports advanced queries)
    Product (Top | Latest)
    From (Start date of tweets YYYY-mm-dd HH:MM:SS)
    To (End date of tweets YYYY-mm-dd HH:MM:SS)
    Frequency (Interval min | hour | day | week | month | year)
  User TL
    Prefix (File prefix for results)
    Botón "Cargar contexto": igual que en Search, pero rellena/limpia
      Users list/From/To/Frequency
    Username(s) (uno o varios usuarios)
    From (Start date of tweets YYYY-mm-dd HH:MM:SS)
    To (End date of tweets YYYY-mm-dd HH:MM:SS)
    Frequency (Sampling frequency - min | hour | day | week | month | year)
  Retweets
    Prefix (Prefix of the file containing the original tweets)
    Min RTs (Minimum number of RTs required to download RTs)
  Comments
    Prefix (Prefix of the file containing the original tweets)
    Min Replies (Minimum number of comments required to download)
  Advanced Comments
    Prefix (Prefix of the file containing the original tweets)
    Min Replies (Minimum number of comments required to download)
    Last (Number of days since the tweet date)
    Frequency (Interval: min | hour | day | week | month | year)

- Tools

  Merge files | Clean data | Location

  Merge files
    Input files (selección múltiple)
    Output file

  Clean data
    Prefix (Prefix of the file with the original tweets)
    Positives (List of included words)
    False positives (List of excluded words)
    Output file

  Location
    Prefix (Prefix of the file with the original tweets)
    acciones:
    - Geocodifica de forma offline (geonamescache) la localización declarada en el perfil de cada usuario presente en {prefix}.csv
    - Identifica país, y si es España, también la Comunidad Autónoma, y ciudad
    resultados:
    - {prefix}_loc.csv con columnas: username, location, country, region, city

- Graphs

  Detect communities | Generate graph | Classify tweets | Visualize graph

  Detect communities
    Prefix (Prefix of the file with comments or RTs)
    Relation (RT | replies | replies_advanced)
    acciones:
    - Construye el grafo dirigido y ponderado de la relación elegida (RT, replies o replies_advanced) y se queda con el componente gigante
    - Calcula el in-degree y out-degree ponderados de cada usuario
    - Detecta comunidades (algoritmo de Louvain/Blondel). El resultado no es determinista: cada ejecución puede dar comunidades distintas, por eso se persiste en disco y no se recalcula en los siguientes pasos
    resultados:
    - {prefix}_{relation}_communities.csv: tabla de comunidades (community, pct_nodes), ordenada de mayor a menor tamaño
    - {prefix}_users_{relation}_communities.csv: comunidad de cada usuario (user, community, weight_indegree, weight_outdegree), ordenada por indegree

  Generate graph
    Prefix (Prefix of the file with comments or RTs)
    Relation (RT | replies | replies_advanced)
    Format (gdf | gexf)
    Include communities (checkbox, usa {prefix}_users_{relation}_communities.csv generado en Detect communities)
    Include locations (checkbox, usa {prefix}_loc.csv generado en Tools > Location)
    resultados:
    - {prefix}_{relation}.gdf o {prefix}_{relation}.gexf, listo para abrir en Gephi o para Visualize graph

  Classify tweets
    Prefix (Prefix of the file with the original tweets)
    Relation (RT | replies | replies_advanced)
    acciones:
    - Cruza {prefix}.csv con {prefix}_users_{relation}_communities.csv para asignar la comunidad del autor a cada tweet
    resultados:
    - {prefix}_{relation}_classified.csv

  Visualize graph
    Select graph (lista de ficheros .gdf/.gexf del proyecto)
    Máximo de etiquetas por comunidad (slider 1-50, por defecto 10)
    Iteraciones de ForceAtlas2 (slider 100-2000, por defecto 300)
    acciones:
    - Muestra el grafo en un visor interactivo embebido en el panel de resultados
      (sigma.js + graphology sobre WebGL, la misma arquitectura que Retina;
      las librerías se cargan de CDN, así que visualizar requiere conexión a internet):
      zoom/pan con el ratón
    - Tooltip al pasar el ratón por un nodo: nombre de usuario y sus atributos, por
      este orden (los vacíos se omiten): create_at_year, user_verified,
      is_blue_verified, verified_type, log_followers_count, log_friends_count,
      log_statuses_count, log_favourites_count, log_listed_count, location_country,
      location_region, location_city
    - El layout ForceAtlas2 se calcula en el navegador en segundos, con los parámetros
      clásicos de Gephi por defecto: gravity 1, scaling 10, sin LinLog ni strong gravity
      (probado sobre panas_RT.gdf: así el hub queda centrado en su halo y las comunidades
      bien separadas; con LinLog o strong gravity el grafo colapsa en una bola).
      Sin límite de nodos (el antiguo límite de 3000 era del layout en Python)
    - Controles de layout dentro del visor, estilo Gephi: Scaling, Gravity, LinLog,
      Disuadir hubs e Iteraciones, con botón Recalcular que continúa desde las
      posiciones actuales (como volver a darle a Run en Gephi)
    - Colorea cada comunidad con un color distinto (las comunidades que no llegan al 2% de los nodos se agrupan como "Otros" en gris); aristas coloreadas según la comunidad del nodo origen (aclaradas hacia blanco) y con grosor según el weight de la arista (escala log)
    - El tamaño de cada nodo depende de su weighted indegree en escala logarítmica (con escala lineal, los hubs de los grafos de RTs aplastan al resto al tamaño mínimo)
    - Etiqueta fija solo para los usuarios más relevantes de cada comunidad (limitado por el slider); el resto de nodos muestran su nombre al pasar el ratón o al hacer zoom
    - Botón "Descargar PNG" dentro del visor: exporta el grafo a PNG en alta resolución al instante (dibujado en un canvas 2D aparte, con etiquetas de tamaño proporcional al nodo)

Notas de implementación:
- Reiniciar la app tras editar cualquier módulo de scripts/ (el hot-reload de Streamlit solo afecta a app.py): usar restart_app.ps1 en la raíz del repo.
- Charts: la lógica de graficado vive en scripts/nb_charts.py y
  nb_charts_profile.py (matplotlib/seaborn, traducción de los antiguos cuadernos
  R utils/charts.R, utils/charts_profile.R y utils/utils_charts.R). El
  envoltorio que usa la app real es scripts/charts.py
  (generate_tweet_charts / generate_user_charts).
- El contexto de las descargas (scripts/context.py) guarda, además
  de las fechas, la query/lista de usuarios, el product y la frequency
  originales, para poder reconstruir el formulario completo con el botón
  "Cargar contexto" al reutilizar un Prefix.
- "Desactivar proyecto" (scripts/projects.py) mueve la carpeta del
  proyecto a data/desactivated/{project} en vez de borrarla.
- Se evaluó una sección "Contents" (Detect entities | Detect frames | Detect sentiment) para análisis de texto de los tweets. Se implementó y se probó (NER con spaCy, diccionarios propios, sentimiento con pysentimiento) pero se eliminó por completo: cada proyecto necesita su propio diccionario hecho a mano, y esa elaboración manual de diccionarios no encajaba en el flujo de la app.
- Antivirus con análisis HTTPS (AVG y similares): twscrape necesita descargar varios scripts desde abs.twimg.com/x.com para calcular el XClientTransactionId que exige Twitter/X. Si el antivirus intercepta el tráfico HTTPS (p. ej. "Web Shield" de AVG) y su certificado de escaneo no cumple con el estándar X.509 (Basic Constraints no marcado como crítico), la descarga falla con "Couldn't get XClientTxId indices script". Solución: desactivar el análisis/escaneo HTTPS del antivirus (no afecta al resto de la protección, solo deja de inspeccionar el contenido del tráfico cifrado) o excluir x.com/twimg.com de ese análisis si el antivirus lo permite.
